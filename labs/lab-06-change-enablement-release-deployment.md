# Lab 06 - Change Enablement, Release, Deployment

## Objectives

- Explain change enablement.
- Distinguish standard, normal, and emergency changes.
- Compare change enablement, release management, and deployment management.
- Build a simple change assessment table.

## Scenario

BrightDesk needs to update the employee portal. One update is a routine monthly patch, one is a new onboarding feature, and one is an urgent fix for a production outage.

## Steps

### 1. Classify Changes

Create a table:

| Change | Type | Reason |
| --- | --- | --- |
| Monthly approved patch | Standard change | Pre-authorized, low risk, repeatable |

Classify:

1. Monthly patch.
2. New onboarding feature.
3. Urgent outage fix.

### 2. Assess a Normal Change

Document:

```text
Description:

```

### 3. Separate Release and Deployment

Write examples:

| Term | Meaning |
| --- | --- |
| Release | Version of service or components made available |
| Deployment | Moving components into live or target environment |

### 4. Create an Emergency Change Rule

Define:

1. Who can authorize.
2. What evidence is required.
3. How users are informed.
4. When post-implementation review happens.
5. How records are updated.

### 5. Apply Guiding Principles

Explain how these apply:

- Focus on value.
- Progress iteratively with feedback.
- Keep it simple and practical.
- Optimize and automate.

## Validation

You should have a change classification table, normal change assessment, release/deployment distinction, and emergency change rule.

## Checkpoint Questions

1. What is the purpose of change enablement?
2. What makes a change standard?
3. Why are emergency changes still controlled?
4. How is deployment different from release?

## Exam Focus

Change questions often test classification and purpose, especially standard, normal, and emergency changes.
